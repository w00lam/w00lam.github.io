## 1. 문제 인식: AI 코드 생성, 그 이상의 자동화가 필요하다

최근 AI 코딩 도구들은 개발 생산성을 혁신적으로 끌어올리고 있습니다. Cursor, Codex, Claude Code, Gemini CLI 등 다양한 도구들이 코드 생성, 디버깅, 문서화에 도움을 주며 개발자들의 일상에 깊숙이 파고들었죠. 저 역시 이러한 도구들을 적극적으로 활용하며 개발 속도 향상을 체감하고 있습니다.

하지만 프로젝트를 진행하면서 한 가지 근본적인 한계를 느끼기 시작했습니다. AI에게 단순히 코드 생성만을 맡기는 방식으로는 팀 전체의 생산성을 극대화하기 어렵다는 점이었습니다. 구체적으로 다음과 같은 문제들이 발생했습니다.

*   **프로젝트마다 작업 방식이 달라짐**: 팀원마다 AI 도구를 활용하는 방식이 제각각이어서 일관된 작업 흐름을 유지하기 어려웠습니다.
*   **문서 품질 및 PR 품질의 불균형**: AI가 생성한 코드나 문서, PR 설명의 품질이 일정하지 않아 추가적인 수작업이 필요했습니다.
*   **작업 프로세스의 비표준화**: 누가 어떤 작업을 하고 있는지, 작업의 진행 상황은 어떤지 파악하기 어려웠고, 작업 범위가 겹치거나 누락되는 경우가 발생했습니다.
*   **반복적인 컨텍스트 입력**: 새로운 대화를 시작할 때마다 프로젝트의 규칙, 컨벤션, 목표 등을 AI에게 반복해서 설명해야 하는 비효율이 있었습니다.

이러한 문제들은 AI가 개별 개발자의 생산성을 높이는 데는 효과적이지만, **팀 단위의 협업 워크플로우 전체를 자동화하고 표준화하는 데는 부족함**이 있다는 것을 깨닫게 했습니다. 저는 코드 생성 자동화를 넘어, 개발 워크플로우 자체를 자동화할 수 있는 시스템이 필요하다고 생각했습니다.

## 2. Agent OS 분석: 문서 중심 개발의 철학

이러한 고민을 하던 중, 저는 Agent OS의 개념을 접하게 되었습니다. Agent OS는 AI 에이전트가 소프트웨어 개발 프로세스에 통합되는 방식에 대한 흥미로운 철학을 제시합니다. 그 핵심은 다음과 같습니다.

*   **문서 우선 개발 (Documentation-First Development)**: 코드를 작성하기 전에 상세한 문서를 통해 요구사항과 설계를 명확히 정의합니다. 이는 AI가 작업을 이해하고 수행하는 데 필요한 컨텍스트를 제공하는 기반이 됩니다.
*   **계획 우선 접근 (Plan-First Approach)**: 무작정 코드를 생성하는 것이 아니라, 명확한 계획과 단계를 수립한 후 AI 에이전트가 이를 따르도록 유도합니다.
*   **AI가 이해하기 쉬운 구조 제공**: AI가 작업을 정확히 수행할 수 있도록, 구조화되고 명확한 형식으로 정보를 제공하는 것을 중요하게 생각합니다.
*   **표준화된 작업 흐름**: 개발 프로세스를 표준화하여 예측 가능하고 일관된 결과를 도출합니다.

Agent OS는 AI 에이전트가 개발 프로세스에 깊이 관여할 때 발생할 수 있는 비일관성, 컨텍스트 부족 등의 문제를 **문서와 계획을 통해 해결**하려는 시도였습니다. 특히, [SDD(Spec Driven Development)에 대한 기술 블로그 포스팅](/posts/ai-sdd-llm-collaboration/)에서 다루었듯이, 명확한 Spec이 AI의 성능을 극대화한다는 관점과 일맥상통하는 부분이 많았습니다. Agent OS는 AI에게 작업을 위임하기 위한 **명확한 지침과 환경을 제공**하는 데 중점을 둡니다.

## 3. Agent OS의 한계: 책임 범위의 차이

Agent OS의 철학은 매우 인상 깊었지만, 실제 제가 필요로 하는 워크플로우 자동화 관점에서는 몇 가지 한계가 있었습니다. 이는 Agent OS의 부족함이라기보다는 **책임 범위의 차이**로 이해하는 것이 더 정확할 것입니다.

Agent OS는 주로 **개발 프로세스 정의**와 **AI 에이전트에게 작업을 위임하기 위한 컨텍스트 제공**에 강점을 가집니다. 하지만 GitHub 기반의 실제 개발 운영 자동화 측면에서는 다음과 같은 기능들을 직접적으로 제공하지 않았습니다.

*   **Commit 생성 책임 없음**: AI가 코드를 생성하더라도, Git Commit 메시지를 생성하고 관리하는 책임은 개발자에게 남아있습니다.
*   **PR 생성 책임 없음**: 작업이 완료된 후 Pull Request를 생성하고, PR 설명을 작성하는 과정은 자동화되지 않습니다.
*   **코드 리뷰 자동화 없음**: 생성된 코드에 대한 자동화된 코드 리뷰 프로세스가 포함되어 있지 않습니다.
*   **이슈 운영 자동화 없음**: GitHub Issue의 생성, 할당, 상태 변경 등 이슈 트래킹 워크플로우를 직접적으로 관리하지 않습니다.
*   **GitHub 운영 자동화는 제공하지 않음**: 브랜치 전략, 머지 정책 등 GitHub 기반의 협업 운영 전반에 대한 자동화 기능은 Agent OS의 직접적인 범위가 아니었습니다.

결론적으로 Agent OS는 AI에게 **"무엇을 어떻게 할지"**를 명확히 지시하는 데 탁월한 아이디어를 제공했지만, **"그 지시를 받은 AI가 실제 개발 환경에서 어떻게 움직여야 할지"**에 대한 구체적인 실행과 자동화까지는 다루지 않았습니다.

## 4. 내가 원했던 것: 코드 생성 자동화가 아닌 개발 워크플로우 자동화

저는 AI가 단순히 코드를 생성하는 것을 넘어, **개발 워크플로우의 전반적인 과정을 자동화**하여 팀의 생산성과 효율성을 극대화하고 싶었습니다. 이는 [AI 에이전트의 Harness(하네스)에 대한 기술적 고찰](/posts/deep-dive-into-ai-agent-harness/)에서 언급했던 에이전트의 실행 환경과 오케스트레이션의 중요성과도 연결됩니다. 즉, AI 에이전트가 개별 작업을 수행하는 것을 넘어, **일련의 작업을 유기적으로 연결하고 관리하는 시스템**이 필요했던 것입니다.

제가 원했던 것은 다음과 같은 목표를 달성하는 시스템이었습니다.

*   **작업의 투명성 확보**: 누가 어떤 작업을 진행하고 있는지, 현재 진행 상황은 어떤지 한눈에 파악 가능해야 합니다.
*   **일관된 품질 유지**: 코드, 문서, PR 등 모든 산출물의 품질이 일정하게 유지되어야 합니다.
*   **반복 작업의 최소화**: AI에게 반복적으로 컨텍스트를 제공하거나, 수동으로 GitHub 작업을 처리하는 비효율을 제거해야 합니다.
*   **개발 프로세스의 표준화**: 팀 전체가 일관된 방식으로 작업하고 협업할 수 있는 표준화된 프로세스를 구축해야 합니다.

이러한 목표를 달성하기 위해 저는 Agent OS의 아이디어를 기반으로, GitHub 워크플로우와 연동되는 **AI 기반 개발 워크플로우 자동화 시스템**을 직접 설계하고 구축하기 시작했습니다.

## 5. 직접 구축한 에이전트 시스템: 워크플로우 체인

제가 설계한 시스템은 여러 AI 에이전트들이 유기적으로 연결되어 개발 워크플로우의 각 단계를 자동화하는 **에이전트 체인(Agent Chain)** 형태로 구성됩니다. 각 에이전트는 특정 역할과 책임을 가지며, 이전 에이전트의 결과물을 받아 다음 단계의 작업을 수행합니다.

![직접 구축한 AI 개발 워크플로우 에이전트 체인](/assets/images/2026-06-08-ai-dev-workflow-automation/custom-agent-workflow-chain.png)

이 워크플로우는 다음과 같은 에이전트들로 구성됩니다.

1.  **Issue Agent**: 사용자 또는 다른 시스템으로부터 요청을 받아 GitHub Issue를 생성하고, 초기 분석을 통해 필요한 정보를 정리합니다.
2.  **Task Agent**: 생성된 Issue를 기반으로 작업을 더 작은 단위의 Task로 분해하고, 각 Task에 대한 상세 계획을 수립합니다. 이 과정에서 [SDD(Spec Driven Development)](/posts/ai-sdd-llm-collaboration/)의 Spec을 활용하여 명확한 지침을 생성합니다.
3.  **Implementation Agent**: Task Agent가 정의한 상세 계획에 따라 실제 코드를 생성하거나 수정합니다. 이 에이전트는 코드 컨벤션, 아키텍처 가이드라인 등을 준수하여 고품질의 코드를 생성하는 데 집중합니다.
4.  **Commit Agent**: Implementation Agent가 생성한 코드를 기반으로 Git Commit 메시지를 자동으로 생성합니다. 이 메시지는 프로젝트의 커밋 컨벤션을 따르며, 변경 사항의 의도와 내용을 명확하게 요약합니다.
5.  **PR Agent**: 작업이 완료되면 Pull Request를 자동으로 생성하고, PR 템플릿에 맞춰 변경 사항, 리뷰 요청 사항, 관련 이슈 등을 포함한 상세 설명을 작성합니다.
6.  **Review Agent**: 생성된 PR에 대해 자동화된 코드 리뷰를 수행합니다. 코드 품질, 잠재적 버그, 보안 취약점, 컨벤션 준수 여부 등을 검토하고 피드백을 제공합니다.
7.  **Documentation Agent**: 코드 변경 사항이나 새로운 기능에 대한 문서를 자동으로 업데이트하거나 생성합니다. API 문서, 사용자 가이드, 기술 명세 등을 포함할 수 있습니다.

이러한 에이전트 체인을 통해 저는 **요구사항 정의부터 코드 생성, 버전 관리, 코드 리뷰, 문서화에 이르는 개발 워크플로우 전반을 자동화**할 수 있었습니다. 각 에이전트가 독립적으로 작동하면서도 전체 워크플로우의 일관성을 유지하는 것이 핵심입니다.

## 6. `AGENTS.md` 활용: 반복 프롬프트 문제 해결

Agent OS의 철학에서 영감을 받아, 저는 `AGENTS.md`라는 파일을 도입하여 **반복적인 컨텍스트 입력 문제**를 해결했습니다. 이 파일은 프로젝트의 모든 AI 에이전트가 공유하는 **Single Source of Truth** 역할을 합니다.

![AGENTS.md를 중심으로 한 컨텍스트 흐름](/assets/images/2026-06-08-ai-dev-workflow-automation/agents-md-context-flow.png)

`AGENTS.md`에는 다음과 같은 내용들이 포함됩니다.

*   **프로젝트 규칙**: 프로젝트의 전반적인 목표, 기술 스택, 아키텍처 원칙 등
*   **브랜치 전략**: Git Flow, GitHub Flow 등 프로젝트가 따르는 브랜치 전략
*   **커밋 컨벤션**: Conventional Commits 등 커밋 메시지 작성 규칙
*   **PR 규칙**: Pull Request 작성 시 포함되어야 할 내용, 리뷰어 지정 규칙 등
*   **문서 규칙**: 문서화 표준, 스타일 가이드 등

각 에이전트는 작업을 시작하기 전에 `AGENTS.md` 파일을 참조하여 프로젝트의 최신 규칙과 컨벤션을 학습합니다. 이를 통해 저는 매번 AI에게 동일한 정보를 반복해서 입력할 필요 없이, **`AGENTS.md` 파일만 업데이트하면 모든 에이전트의 행동을 일관되게 제어**할 수 있게 되었습니다. 이는 AI와의 협업에서 발생하는 컨텍스트 관리의 비효율성을 크게 줄여주었습니다.

## 7. Agent OS vs 직접 구축 시스템 비교

Agent OS와 제가 구축한 시스템은 AI 기반 개발 프로세스를 개선한다는 공통점을 가지지만, 책임 범위와 목표에서 명확한 차이를 보입니다. 다음 표는 두 시스템의 주요 비교 항목을 보여줍니다.

![Agent OS vs 직접 구축 시스템 비교](/assets/images/2026-06-08-ai-dev-workflow-automation/agent-os-vs-custom-system.png)

이 비교를 통해 알 수 있듯이, Agent OS는 **AI에게 작업을 위임하기 위한 "지침서"** 역할에 가깝다면, 제가 구축한 시스템은 그 지침서를 바탕으로 **실제 개발 워크플로우를 "실행하고 관리"하는 자동화된 파이프라인**에 가깝습니다. 두 시스템은 경쟁 관계가 아니라, 서로 다른 책임 범위를 가지며 상호 보완적으로 활용될 수 있습니다.

## 8. 향후 발전 방향: 더 지능적인 워크플로우를 향하여

현재 구축된 시스템은 기본적인 개발 워크플로우 자동화를 성공적으로 수행하고 있지만, 앞으로 더욱 지능적이고 효율적인 시스템으로 발전시킬 계획입니다. 몇 가지 구체적인 발전 방향은 다음과 같습니다.

*   **자동 Label 생성**: Issue 내용 분석을 통해 적절한 GitHub Label을 자동으로 부여하여 이슈 분류의 효율성을 높입니다.
*   **GitHub Project 자동 등록**: 생성된 Issue나 PR을 GitHub Project의 칸반 보드에 자동으로 등록하고, 진행 상황에 따라 상태를 자동으로 변경합니다.
*   **배포 후 Health Check 및 Slack 알림**: 배포가 완료된 후 서비스의 Health Check를 자동으로 수행하고, 이상 발생 시 Slack을 통해 팀에 즉시 알림을 전송합니다.
*   **배포 검증 자동화**: 배포된 기능에 대한 간단한 E2E 테스트나 통합 테스트를 자동으로 실행하여 배포의 안정성을 검증합니다.
*   **피드백 루프 강화**: Review Agent의 피드백을 Implementation Agent가 학습하여 코드 생성 품질을 지속적으로 개선합니다.

이러한 기능들을 추가함으로써, AI 에이전트가 개발 워크플로우의 더 많은 부분을 자율적으로 처리하고, 개발자는 더욱 고차원적인 문제 해결에 집중할 수 있도록 만들고자 합니다.

## 9. 결론: Agent OS는 경쟁 대상이 아니라 기반 아이디어였다

이번 경험을 통해 저는 Agent OS가 제시하는 **문서 중심, 계획 우선 개발**이라는 철학이 AI 시대의 개발 워크플로우를 설계하는 데 있어 매우 중요한 기반 아이디어가 된다는 것을 깨달았습니다. Agent OS는 AI에게 명확한 컨텍스트를 제공하는 방법을 제시했지만, 실제 GitHub 기반의 개발 운영 자동화까지는 다루지 않았습니다.

저는 이 아이디어를 확장하여, [AI 에이전트의 Harness(하네스)에 대한 기술적 고찰](/posts/deep-dive-into-ai-agent-harness/)에서 다루었던 에이전트의 실행 환경과 오케스트레이션 개념을 접목하여 **요구사항 정의부터 코드 생성, 버전 관리, 코드 리뷰, 문서화에 이르는 전반적인 개발 워크플로우를 자동화하는 시스템**을 구축하게 되었습니다.

AI는 단순히 코드를 생성하는 도구를 넘어, **개발 프로세스 자체를 혁신하는 강력한 협업 파트너**가 될 수 있습니다. 이를 위해서는 AI가 효과적으로 작동할 수 있는 명확한 지침, 표준화된 워크플로우, 그리고 유기적으로 연결된 에이전트 시스템의 설계가 필수적입니다. "느리게 가는 것이 빠르게 가는 것"이라는 말처럼, 무작정 코드 생성에만 의존하기보다 **명세 설계와 워크플로우 자동화에 대한 깊은 고민**이 AI 시대 개발 생산성 향상의 핵심임을 다시 한번 강조하고 싶습니다.
